\section{校准产物格式}
\label{sec:app-calib-schema}
% ============================================================

离线校准阶段产出的 JSON 文件遵循以下 schema：

\begin{lstlisting}[
  basicstyle=\small\ttfamily,
  frame=single,
  breaklines=true,
  numbers=left,
  numberstyle=\tiny,
  caption={校准产物 JSON schema},
  label={lst:calib-schema}
]
{
  "model_name": "Qwen/Qwen2.5-1.5B-Instruct",
  "calib_method": "kl_selected",
  "num_layers": 28,
  "layers": {
    "0": {
      "k_scale": <float32>,
      "v_scale": <float32>,
      "inv_tau": [<float32>, ...],  // per-head, length = num_heads
      "k_zero_point": <float32>,    // optional, asymmetric only
      "v_zero_point": <float32>     // optional, asymmetric only
    },
    "1": { ... },
    ...
  },
  "metadata": {
    "quant_bits": 8,
    "calibration_samples": 128,
    "calibration_seq_len": 2048,
    "timestamp": "2026-xx-xx"
  }
}
\end{lstlisting}

其中：
\begin{itemize}[nosep]
  \item \code{k\_scale} / \code{v\_scale}：逐层对称量化的缩放因子，始终以 float32 存储。
  \item \code{inv\_tau}：逐头温度校正因子 $\tau^{-1}$，用于在注意力 softmax 前调节 logits 的分布锐度。
  \item \code{k\_zero\_point} / \code{v\_zero\_point}：仅在 KIVI 风格非对称量化模式下出现。
\end{itemize}

\section{INT4 量化噪声与序列长度交互效应的形式化推导}
\label{sec:app-sqnr-derivation}
% ============================================================

本节提供正文第~\ref{subsec:exp-int4-limitations}~节中量化步长与序列长度交互效应的完整推导。

Needle-in-a-Haystack 任务的检索成功条件可形式化为：
目标位置 $t$ 的 attention logit 与最强干扰位置的差值
$\delta = z_t - \max_{i \neq t} z_i$
（其中 $z_i = \bq^\top \bk_i / \sqrt{d_k}$）
需在量化后仍保持正值，使 softmax 将最大概率分配给目标位置。

当 Key 矩阵经 INT4 量化后，每个 logit $z_i$ 受到量化噪声扰动 $\epsilon_i$，
其方差约为：
\begin{equation}
  \text{Var}(\epsilon_i) \approx \|\bq\|^2 \cdot \sigma_{q,4}^2 / d_k
\end{equation}
其中 $\sigma_{q,4}^2 = \Delta_4^2 / 12$ 为 INT4 量化的逐元素噪声方差。

随序列长度 $n$ 增大，$\max_{i \neq t} (z_i + \epsilon_i)$ 的期望值
以 $O(\sqrt{2 \ln n} \cdot \sigma_\epsilon)$ 的速率增长（极值统计理论），
而目标信号 $\delta$ 不随 $n$ 增大。
因此存在临界序列长度 $n^*$，满足：
\begin{equation}
  \delta \approx \sqrt{2 \ln n^*} \cdot \sigma_\epsilon
\end{equation}
当 $n > n^*$ 时，量化噪声以高概率淹没目标信号，检索失败。

由于 INT4 的逐元素噪声方差 $\sigma_{q,4}^2 / \sigma_{q,8}^2 = (255/15)^2 = 289$，
INT4 的 $n^*$ 远小于 INT8，从理论上解释了 INT4 在 1.5B 模型上
从短上下文到 32K 的 Needle 通过率急剧下降趋势。

    使用修正后代码重跑的 KIVI INT4 cs$=$1 PPL 为 10{,}332
    （旧版本为 9442），与 INT4-RoleAlign cs$=$1 的 $>$10{,}000 在同一量级，
    确认 per-channel K Scale 在 cs$=$1 下的崩溃是所有 INT4 非对称方法的共同边界。
    \item[$\dagger$] INT8-ours 的 cs=128 PPL 为 8.95（低于 FP16 的 9.31），
    因为两者使用不同的 max\_length/stride 配置。
    cs=1 时退化 +3.6\% 相对 cs=128，说明全量化下存在少量超出校准范围的激活值裁剪，
    自适应保护机制部分缓解了此问题。
    \item 注：括号内为相对 FP16 的 PPL 退化率（INT8-ours 的退化率相对自身 cs=128）。
    cs = chunk\_size。FP16 不含量化噪声，PPL 不随 cs 变化（预期内）。
  \end{tablenotes}
  \end{threeparttable}
\end{table}

\paragraph{结果分析。}

\textbf{per-channel K Scale 在 cs=1 下的共同崩溃}：
INT4-RoleAlign 和 KIVI-style 在 cs=1 下均出现灾难性 PPL 退化
（RA $>$10{,}000，KIVI 10{,}332）。
两者共享同一根因：per-channel K Scale 在 cs=1 下
仅基于首个 token 的 K 值建立，scale 范围极窄（覆盖单 token 的 channel 分布），
后续 decode tokens 的 K 值几乎 100\% 超出 clamp 范围，
导致注意力权重计算严重失真。
INT8-ours 的退化相对温和（+3.6\%），因为 INT8 的 256 个量化级别
为 scale 范围外的值提供了更大的容错空间。

\textbf{对称 vs 非对称在 cs=1 下的行为差异}：
KIVI-style 额外受到 per-token V Scale 的影响。
当 cs=1 时每个 token 的 V Scale 独立计算，
单 token 统计量方差极大，V 量化噪声叠加 K 量化噪声导致更严重的崩溃。
INT4-RoleAlign 的 V Scale 同样 per-token，但 BA percentile 校准
对 V 使用了 99.9\% 而非 100\% 的裁剪阈值，部分缓解了极端值影响。

\textbf{修正说明}：先前版本的附录报告 INT4-RoleAlign 在 cs=1 下 PPL=9.49
并据此声称"BA-guided percentile 在逐 token 粒度下仍然有效"。
该数据经溯源确认来自评测脚本缺陷期间
（eval\_ppl.py 缺少 int4\_ours\_asym 分支，静默回退 INT8 路径），
实际评测的是 INT8 量化而非 INT4-RoleAlign。
更正后的数据显示 INT4-RoleAlign 在 cs=1 下同样崩溃，
表明 per-channel K Scale 在逐 token 评测粒度下的固有局限性
是所有使用该量化轴策略的方法的共同边界，而非 KIVI-style 的特有缺陷。
上述观察仅在 Qwen2.5-1.5B 单模型、单种子条件下验证
（见第~\ref{sec:conclusion-limitations}~节）。

\section{逐头温度校正机制详细推导}
\label{sec:app-invtau-detail}

本节给出第~\ref{sec:ch3-invtau}~节中逐头温度校正（$\tau^{-1}$）的完整理论推导，
包括量化导致注意力平坦化的分析、温度校正的数学描述以及 Q 预缩放等价性。

\subsection{量化导致注意力分布平坦化的原因分析}

量化过程引入的舍入误差和裁剪误差作用于 Key 矩阵 $\bK$ 后，
会导致注意力 logits $\bq\hat{\bK}^T / \sqrt{d_k}$ 相对于参考值 $\bq\bK^T / \sqrt{d_k}$ 发生偏移。
由于量化误差的随机性特点，同一 Query 向量与不同 Key 向量的点积受到近似均匀的噪声扰动，
导致 logits 之间的差异被缩小，softmax 输出的概率分布趋于平坦化。

这一现象可以从信息论角度理解：
当注意力分布从尖锐（低熵）变为平坦（高熵）时，
模型对关键位置的聚焦能力下降，
等价于在注意力层引入了信息丢失。
对于需要精确捕捉长距离依赖的任务（如大海捞针检索、长文本分类等），
这种平坦化效应尤为有害。

不同层和不同注意力头受量化影响的程度各异：
浅层头通常关注局部模式（如相邻 token），其注意力分布本身较为分散，受平坦化影响相对较小；
深层头则可能负责全局信息汇聚，具有高度集中的注意力分布，量化导致的平坦化对其影响更为显著。
这一观察构成了逐头（per-head）温度校正的核心动机。

\subsection{温度校正的数学描述}

引入逐头温度校正因子 $\tau^{-1}_{l,h}$（inverse temperature），
校正后的注意力计算定义为：
\begin{equation}
  \bm{p}_{\text{corrected}} = \softmax\!\left(\frac{\bq\hat{\bK}^T \cdot \tau^{-1}_{l,h}}{\sqrt{d_k}}\right)
  \label{eq:app-corrected-attn}
\end{equation}

温度校正因子的物理含义如下：
\begin{itemize}
  \item $\tau^{-1} > 1$（等效温度 $\tau < 1$）：
    缩放后的 logits 绝对值增大，softmax 输出更加尖锐（锐化效应），
    可补偿量化导致的分布平坦化；
  \item $\tau^{-1} < 1$（等效温度 $\tau > 1$）：
    缩放后的 logits 绝对值减小，softmax 输出更加平滑（平滑化效应），
    适用于量化后注意力分布过度集中的情形；
  \item $\tau^{-1} = 1$：
    不施加任何校正，退化为标准量化基线。
\end{itemize}

\subsection{Q 预缩放等价性}

在实际实现中，温度校正因子 $\tau^{-1}$ 并不直接作用于 logits，
而是通过预缩放 Query 向量来等价实现。这一设计的数学基础如下：

\begin{equation}
\begin{aligned}
  \text{Attn}(\bQ \cdot \tau^{-1},\; \hat{\bK},\; \hat{\bV})
  &= \softmax\!\left(\frac{(\bQ \cdot \tau^{-1})\hat{\bK}^T}{\sqrt{d_k}}\right) \hat{\bV} \\[6pt]
  &= \softmax\!\left(\frac{\bQ \hat{\bK}^T \cdot \tau^{-1}}{\sqrt{d_k}}\right) \hat{\bV}
\end{aligned}
\label{eq:app-q-prescale}
\end{equation}

式~\eqref{eq:app-q-prescale} 表明，
将 $\tau^{-1}$ 作用于 $\bQ$ 与将其作用于 $\bQ\hat{\bK}^T$ 在数学上完全等价，
因为标量乘法满足结合律。
这一等价性使得温度校正可以在 Query 投影阶段一次性完成，具有以下实现优势：

\begin{enumerate}[label=(\arabic*)]
  \item \textbf{无需修改注意力核函数签名}：
    Q 预缩放在进入 Triton 融合核函数之前完成，
    核函数内部的 $\bQ\hat{\bK}^T$ 计算和 online softmax 逻辑无需任何修改，
    从而避免了核函数的代码膨胀和多版本维护成本。
  \item \textbf{逐头独立缩放}：
    $\tau^{-1}$ 存储为形状 $[L, H_q]$ 的张量，
    在 prefill 和 decode 阶段通过广播（broadcast）作用于 $\bQ$ 的对应头维度：
    \begin{equation}
      \bQ'_{l,h,:,:} = \bQ_{l,h,:,:} \cdot \tau^{-1}_{l,h}
      \label{eq:app-q-prescale-impl}
    \end{equation}
    其中 $l$ 为层索引，$h$ 为头索引。
  \item \textbf{Prefill 阶段无额外 kernel launch}：
    Q 预缩放是逐元素乘法，可与 Query 投影的输出张量原地（in-place）操作融合，
    不引入额外的 GPU kernel 启动开销。
\end{enumerate}
