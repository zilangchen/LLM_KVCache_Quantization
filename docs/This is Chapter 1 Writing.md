### 2.2 小节职责

这一小节只承担四件事：

3. 把 `KV Cache` 显存压力写成一个明确、可感知的部署瓶颈，并顺手点出仅靠 `PagedAttention` 这类管理优化并不能降低绝对存储量。
4. 自然引出量化压缩的现实价值，尤其是 `INT8/INT4` 在上下文长度与并发批次上的直接收益，为 `1.2` 的问题提出提供入口。当前 `main(6).pdf` 里的这条主骨架是可复用的，不需要推倒重写。

### 2.3 本节不做什么

这一节必须明确克制，不承担以下内容：

### 2.4 最优段落结构

建议 `1.1` 固定为 **4 个自然段 + 1 条轻量背景公式 + 0 张图 + 0 张表**。

**第 1 段：长上下文发展与部署压力。**  
先从应用趋势写起：大语言模型参数规模和上下文窗口持续扩展，长文本理解、复杂推理与多轮生成能力增强，但推理阶段的显存与延迟开销也随之抬升。这里可以保留当前 PDF 中“从 4K 到 128K tokens、代表性模型不断推进长上下文能力”的基本表述，但要把重点压在**部署压力上升**，而不是模型能力陈列上。

**第 2 段：`KV Cache` 的作用与代价。**  
接着解释自回归解码为何会缓存历史 `K/V`，并保留一句 `KV Cache` 使单步重复计算从 `$O(n^2)$` 降为 `$O(n)$` 的背景说明。这里要保留“它是核心加速手段”的正面评价，同时马上接一句“代价是显存占用随序列长度线性增长”，让本节焦点稳定落在**存储压力**，而不是计算原理。当前 PDF 这一段主体可以直接复用。

**第 3 段：轻量背景公式 + 管理优化边界。**  
这里保留一条轻量背景公式 `$M_{KV} = 2 \\times L \\times H_{kv} \\times d_k \\times S \\times b$`，只承担“显存随层数、`KV` 头数、头维度、序列长度和存储精度增长”的直观提示功能，不做数值推导，也不展开示例计算。变量解释控制在一句话内即可，详细推导和数值例子明确留给第二章 `2.2`。紧接着保留当前 PDF 中那句很有用的边界说明：`PagedAttention` 之类方法改善显存管理与碎片，但不减少 `KV Cache` 的绝对存储量。这样，`1.1` 与 `2.2` 的接口会非常清楚。当前 PDF 与 Chapter 2 施工文档都支持这种写法。

**第 4 段：量化压缩的现实价值 + 向 1.2 过渡。**  
最后一段收束到量化：把 `FP16` 压到 `INT8/INT4` 可把 `KV Cache` 显存降到原来的 `1/2` 与 `1/4`，这使同一硬件预算下支持更长上下文或更高并发成为可能。这里不讨论“怎么量化”，只强调“为什么值得做”。段末建议加一句轻度过渡到 `1.2`：  
**“然而，压缩率的提升并不自动等于推理行为的稳定保持，尤其是在更具吸引力的低比特区间内，如何在显存收益与模型可用性之间取得平衡，构成本文进一步需要回答的问题。”**  
这句过渡可以把读者自然送到 `1.2`，但还没有越界到 `softmax` 或行为失真的正式动机。
